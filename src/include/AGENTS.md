# AGENTS.md: `src/include` — internal top-level headers and global state

This document orients AI agents and human contributors working in
`src/include`. It assumes you have already read the top-level
[`AGENTS.md`](../../AGENTS.md) — the golden rules (prefix conventions,
`pmix_config.h`-first include order, constant-on-the-left comparisons,
brace-everything, `#define` logical macros to `0`/`1` and test with
`#if`, warning-free under `--enable-devel-check`), the
**thread-safety / progress-thread model and the caddy pattern**, the
wire-format / backward-compatibility rules, and the copyright-header
requirement all apply here and are not repeated. This file covers what
is specific to `src/include`.

Like `src/class` and `src/client`, **`src/include` is not an MCA
framework.** There are no components and no selection logic. It is the
home of the library's lowest-level internal headers — the ones that sit
*below* almost everything else in the include graph — plus the single
translation unit that instantiates the library's global state and the
`PMIX_CLASS_INSTANCE` definitions for a large family of internal
objects.

Do not confuse this directory with the top-level `include/` (no `src/`
prefix): that holds the **public** installed headers (`pmix.h`,
`pmix_common.h`, `pmix_server.h`, …). `src/include` is **internal** —
its headers are installed only so that other PMIx-internal consumers
(and the occasional tightly-coupled companion build) can reach them; user
code must never include them.

## The single most important thing: generated vs. source

Several of the files sitting in this directory on a configured/built
tree are **generated** and must never be hand-edited or reviewed as
source. Editing them is wasted work — they are overwritten on the next
build or `configure` run, and most are in `.gitignore`. Know which is
which before you touch anything:

| File(s) | Produced by | Source of truth |
|---------|-------------|-----------------|
| `pmix_config.h` | `./configure` | `pmix_config.h.in` (itself generated by `autogen.pl`) |
| `pmix_config.h.in` | `autogen.pl` / autoheader | `configure.ac`, `config/*.m4` |
| `pmix_dictionary.{c,h}` | `contrib/construct_dictionary.py` (BUILT_SOURCES) | `include/pmix_common.h.in`, `include/pmix_deprecated.h`, `src/common/pmix_attributes.c` |
| `pmix_event_strings.{c,h}` | `contrib/construct_event_strings.py` | the `PMIX_*` event codes in the public headers |
| `pmix_frameworks.{c,h}` | `autogen.pl` | the set of frameworks under `src/mca/` |
| `Makefile`, `Makefile.in` | autotools | `Makefile.am` |

The **hand-written source** actually maintained here — and the subject
of the rest of this document — is:

```
pmix_globals.h   pmix_globals.c        <- global state + internal object zoo
pmix_config_top.h  pmix_config_bottom.h  <- config.h wrappers (compiler/portability)
pmix_types.h                             <- libevent + byte-order + portability typedefs
pmix_stdint.h                            <- fixed-width / pointer-sized integer types
pmix_socket_errno.h                      <- trivial errno alias
pmix_hash_string.h                       <- string-hash macros
pmix_atomic.h  pmix_stdatomic.h          <- atomic primitives / typedefs (C11)
pmix_prefetch.h                          <- branch-prediction & prefetch hints
pmix_portable_platform.h                 <- one-line shim
pmix_portable_platform_real.h            <- VENDORED GASNet header — FROZEN, do not edit
```

## `pmix_globals.{h,c}` — the heart of this directory

This is where nearly all of the developer-relevant code lives; the rest
of the directory is portability plumbing.

### `pmix_globals.h`

Declares the process-wide singleton `pmix_globals` (of type
`pmix_globals_t`, bottom of the header) — event base, my proc identity,
uid/gid, hostname/aliases, the notifications hotel, IOF controls, the
`nspaces` list, topology, and the attribute `keyindex`. It is
**instanced and initialized in `src/runtime/pmix_init.c`**, not here.

It also defines the bulk of the library's internal object *types* — the
"caddy zoo" that the progress-thread model runs on:

- **Thread-shift machinery.** `PMIX_THREADSHIFT(r, c)` and
  `PMIX_THREADSHIFT_DELAY(r, c, t)` are defined here (near line 756).
  Every struct passed to them **must** carry a `pmix_event_t ev` field
  as its first-touched member — this is the caddy contract from the
  top-level guide. The canonical caddies are `pmix_shift_caddy_t`,
  `pmix_cb_t`, `pmix_query_caddy_t`, `pmix_notify_caddy_t`,
  `pmix_server_caddy_t`, and `pmix_server_trkr_t`; their `PMIX_CLASS_INSTANCE`
  constructors/destructors live in `pmix_globals.c`.
- **Peer / namespace model.** `pmix_peer_t`, `pmix_namespace_t`,
  `pmix_rank_info_t`, `pmix_personality_t`, `pmix_epilog_t`. Read the
  field comments — several encode hard-won invariants (e.g. the
  `pmix_peer_t.gds` per-peer GDS override, the `local_app_fini_fired`
  re-fire guard, and the collective-tracker `departed`-vs-`local_cbs`
  identity rule at ~line 604).
- **The wire command enum.** `pmix_cmd_t` and the `PMIX_*_CMD` values
  (lines ~228-271). **These are wire values** — the same
  interoperability rules as `bfrops` apply: only ever *append* new
  commands at the end (see how `PMIX_GDS_FALLBACK_CMD` was added), never
  insert, remove, or renumber. `pmix_command_string()` (in
  `src/util/`) must be kept in sync.
- **Static-init macros.** Structs that can be stack-allocated ship a
  `_STATIC_INIT` companion (e.g. `PMIX_IOF_FLAGS_STATIC_INIT`,
  `PMIX_KEYINDEX_STATIC_INIT`). If you add a field to such a struct,
  update **both** the `PMIX_CLASS_INSTANCE` constructor in
  `pmix_globals.c` **and** the static-init macro, or the two
  initialization paths silently diverge.
- **Inline key classifiers.** `pmix_check_node_info()`,
  `pmix_check_app_info()`, `pmix_check_session_info()`,
  `pmix_check_special_key()`, `pmix_check_local()` are `static inline`
  lookup tables. When a new node/app/session-scoped attribute is added
  to the standard, the relevant table here usually needs a new entry —
  it is easy to forget and there is no compile-time check.

### `pmix_globals.c`

Its banner says it all: *"THIS FILE IS INCLUDED SOLELY TO INSTANTIATE
AND INIT/FINALIZE THE GLOBAL CLASSES."* It is almost entirely
`PMIX_CLASS_INSTANCE(...)` definitions with their con/destructors, plus:

- `pmix_execute_epilog()` / `dirpath_destroy()` / `dirpath_is_empty()` —
  the post-termination cleanup that unlinks files and recursively removes
  directories registered in a `pmix_epilog_t`. This is the one place
  here that does real filesystem work; tread carefully (it calls
  `unlink`/`rmdir`).
- Thin libevent wrappers `pmix_event_assign()` / `pmix_event_new()`
  (the `pmix_event_*` macros they back live in `pmix_types.h`).
- `pmix_dstor_new_tma()` / `pmix_dstor_release_tma()` and the
  `keyindex` con/destructor — note these are **TMA-aware** (they honor a
  `pmix_tma_t` custom allocator so the objects can live in shared
  memory for the `gds/shmem2` component). If you touch object memory
  management here, preserve the TMA path.

When you add an internal object type, the pattern is: declare the struct
+ `PMIX_CLASS_DECLARATION` in `pmix_globals.h`, and add the matching
`PMIX_CLASS_INSTANCE` + con/destructor in `pmix_globals.c`. Zero every
field in the constructor and free/RELEASE every owned field in the
destructor — the existing definitions are the reference.

## Config plumbing: `pmix_config_top.h` / `pmix_config_bottom.h`

These two are **hand-written source** that the *generated* `pmix_config.h`
`#include`s at its very top and very bottom respectively (that is the
whole reason `pmix_config.h.in` exists as a wrapper). They are
Open-MPI-lineage portability boilerplate:

- **`pmix_config_top.h`** just `#undef`s the autoconf `PACKAGE_*` /
  `HAVE_CONFIG_H` macros so that including a third-party project's
  `config.h` alongside PMIx's does not collide.
- **`pmix_config_bottom.h`** is the important one: it defines the
  `BEGIN_C_DECLS`/`END_C_DECLS` guards, **all** the
  `__pmix_attribute_*__` compiler-attribute wrappers (this is the file
  the top-level guide points you at when it says "use the
  `__pmix_attribute_*__` macros"), the `asprintf`/`snprintf` fallbacks,
  `PMIX_PATH_MAX`/`PMIX_MAXHOSTNAMELEN`, and the sockaddr/AF_* shims.
  Code under `#if PMIX_BUILDING` is internal-only; the `#else` half is
  what leaks into consumers that include a PMIx `config.h`.

## Portability & type headers

- **`pmix_stdint.h`** — Open-MPI-derived. Includes `<stdint.h>` and
  fills gaps: 128-bit types (`pmix_int128_t`), pointer-sized
  `intptr_t`/`uintptr_t` fallbacks keyed off `SIZEOF_VOID_P`, and
  `PRIsize_t`. Effectively frozen portability code — change only for a
  real new-platform need.
- **`pmix_types.h`** — pulls in libevent (`<event.h>`, `<event2/thread.h>`)
  and defines the `pmix_event_*` macro veneer over it (`pmix_event_add`,
  `pmix_event_assign`, `pmix_event_evtimer_set`, …) that the whole
  library uses instead of calling libevent directly. Also holds
  byte-order helpers (`pmix_hton64`/`pmix_ntoh64`, `pmix_htonsizet`) and
  the `pmix_ptr_t` union. If you need to swap the event backend or add an
  event wrapper, this is the choke point.
- **`pmix_socket_errno.h`** — trivial: `#define pmix_socket_errno errno`
  (a Windows-portability vestige). Nothing to see.
- **`pmix_hash_string.h`** — `PMIX_HASH_STR` / `PMIX_HASH_STRLEN`
  (Jenkins one-at-a-time hash). Used by `src/util/pmix_hash.c`.

## Atomics: `pmix_atomic.h` / `pmix_stdatomic.h`

Both are hand-written and built on C11 `<stdatomic.h>`, and both are
**widely depended upon** (`pmix_atomic_check_bool` alone appears in ~30
files) — changing a typedef or macro signature ripples across the tree.

- **`pmix_stdatomic.h`** — the atomic *typedefs* (`pmix_atomic_int32_t`,
  `pmix_atomic_bool_t`, `pmix_atomic_size_t`, …, thin `_Atomic` wrappers)
  plus convenience macros. Included by `src/threads/pmix_threads.h`,
  the PTL headers, `pmix_show_help.h`, and the progress-thread code.
- **`pmix_atomic.h`** — memory barriers (`pmix_atomic_wmb`/`rmb`, used by
  `PMIX_POST_OBJECT`/`PMIX_ACQUIRE_OBJECT` in `pmix_threads.h`) and the
  32-bit fetch-op inline family (`pmix_atomic_fetch_add_32`, …, used by
  `pmix_shmem.c`). Includes `pmix_stdatomic.h`.

**Design note — the `__atomic_*` builtins in `pmix_stdatomic.h` are
deliberate, not a bug.** Four of its convenience macros
(`pmix_atomic_store_int`, `pmix_atomic_load_int`, `pmix_atomic_fetch_add`,
`pmix_atomic_test_and_set`) use the GCC/Clang `__atomic_*` builtins
rather than C11 `<stdatomic.h>` — this is the *only* place in the tree
that does. The reason: their callers operate atomically on **ordinary,
non-`_Atomic`-qualified** globals (`pmix_show_help_enabled`,
`server_init_cntr`/`client_init_cntr`/`tool_init_cntr` — all plain
`int` — and `pmix_globals.init_called`, a plain `bool`). The `__atomic_*`
builtins accept a pointer to a non-atomic object; the C11 `atomic_*`
functions do not. Do **not** naively "port" these to C11 — a correct
port would require `_Atomic`-qualifying all of those globals across
`src/server`, `src/client`, `src/tool`, and `src/util`. Every compiler
PMIx actually targets (GCC, Clang, icc/icx, nvhpc, armclang) provides
`__atomic_*`, so the practical portability exposure is nil. Note the
memory ordering differs on purpose: `pmix_atomic_fetch_add` here is
`SEQ_CST` (init/finalize counters), whereas the differently-named
`pmix_atomic_fetch_add_32` in `pmix_atomic.h` is relaxed (shared-memory
refcounts in `pmix_shmem.c`).

## Platform detection: `pmix_portable_platform*.h`

- **`pmix_portable_platform.h`** — a one-line PMIx-owned shim that just
  `#include`s the real header. Safe to touch; nothing in it.
- **`pmix_portable_platform_real.h`** — **VENDORED, FROZEN.** This is
  GASNet's `portable_platform.h` (Dan Bonachea / UC Regents), cloned
  verbatim at `PLATFORM_HEADER_VERSION 22`. Its own license block
  forbids modifying the version-handshake section. **Do not hand-patch
  it.** The correct maintenance action is to re-sync a newer version
  from GASNet upstream. It only feeds the platform string printed by
  `pmix_info`. Note it depends on PMIx supplying `SIZEOF_VOID_P` (from
  `pmix_config.h`) before inclusion, and it contains `#error` landmines
  that hard-stop the build on an unrecognized arch/endianness.

## `pmix_prefetch.h`

`PMIX_LIKELY`/`PMIX_UNLIKELY` (over `__builtin_expect`) and
`PMIX_PREFETCH` (over `__builtin_prefetch`). **`PMIX_PREFETCH` silently
expands to nothing** when `PMIX_C_HAVE_BUILTIN_PREFETCH` is `0` — and it
*is* `0` in at least some configured trees (e.g. this one), so it is
frequently a no-op. Corollary: **never put a side effect inside a
`PMIX_PREFETCH(...)` argument** — the arguments are not evaluated when
disabled. `PMIX_LIKELY`/`PMIX_UNLIKELY` degrade safely (they still
evaluate their expression).

## Include-order role

These headers sit near the bottom of the include graph. `pmix_config.h`
(hence `pmix_config_top.h`/`pmix_config_bottom.h`) must still be the
first include in any `.c` file, exactly as the golden rules say. The
dependency spine within this directory is:

```
consumer → pmix_atomic.h → pmix_stdatomic.h → pmix_stdint.h → pmix_config.h
pmix_globals.h → pmix_types.h (+ event.h) → pmix_config.h
```

## Cleanup history (July 2026 review)

A review of this directory turned up several defects, all now fixed on
`topic/src-include-cleanup`. Recorded here so the same dead code is not
"rediscovered" and re-added:

- **Deleted broken, unused `PMIX_REGATTR_INPUT_FREE`** from
  `pmix_globals.h`. Its unbraced inner `if` closed the wrong block,
  leaving the macro with unbalanced braces (it would not have compiled
  if expanded), and it freed a `PMIx_Argv_copy`'d `char**` with plain
  `free()`. It had zero call sites. The correct attribute-teardown
  reference is `keyindex_destruct()` in `pmix_globals.c`.
- **Deleted the unused `pmix_swap_bytes{2,4,8}` family** from
  `pmix_types.h`. On the common little-endian path the 8-byte macro
  expanded to a nonexistent `hton64` symbol; the whole family had no
  callers. (The live byte-order helpers `pmix_hton64`/`pmix_ntoh64`/
  `pmix_htonsizet` remain.)
- **Fixed a `__GNUC_MINOR` → `__GNUC_MINOR__` typo** in
  `pmix_config_bottom.h` (the missing `__` made the token evaluate to 0).
- **Removed the dead `PMIX_ATOMIC_X86_64` branch** and the stale
  `PMIX_HAVE_ATOMIC_*` doc block from `pmix_atomic.h` (the macro was
  defined nowhere, so the branch never compiled).
- **Removed the unused `pmix_atomic_int_t` / `pmix_atomic_long_t`
  typedefs** from `pmix_stdatomic.h` (they were the header's only
  dependence on the *optional* `atomic_int`/`atomic_long` convenience
  types that `configure` probes as `PMIX_C_HAVE_ATOMIC_CONV_VAR`). The
  `__atomic_*` macros were left as-is — see the design note under
  **Atomics** for why.
- **Removed a stray `pmix_config.h.in~` editor backup** from the working
  tree.

## Building

This directory builds the `noinst` convenience library
`libpmixglobal.la` (the compiled `pmix_globals.c`, `pmix_dictionary.c`,
`pmix_frameworks.c`, `pmix_event_strings.c`) and installs the internal
headers under `$(pmixincludedir)`. The `Makefile.am` is forced
`.NOTPARALLEL` on purpose — `construct_dictionary.py` writes two targets
and must not run twice concurrently. `BUILT_SOURCES` guarantees the
dictionary/event-string files are generated before anything else.

To compile-check a change, build from the repo root:

```sh
make -j$(nproc)
```

If you changed anything the dictionary/event-string generators depend on
(the public attribute/event definitions, or `src/common/pmix_attributes.c`),
those files regenerate automatically. If you edited `Makefile.am`, a
plain `make` regenerates `Makefile`; if you changed `configure.ac` or
`*.m4`, follow the full `autogen.pl` + `configure` procedure from the
top-level guide.

## When adding or modifying code here

- Confirm you are editing **source**, not a generated file (see the table
  at the top). Never hand-edit `pmix_config.h`, `pmix_dictionary.*`,
  `pmix_event_strings.*`, `pmix_frameworks.*`, or the `Makefile`s.
- New internal object type → struct + `PMIX_CLASS_DECLARATION` in
  `pmix_globals.h`, `PMIX_CLASS_INSTANCE` + con/destructor in
  `pmix_globals.c`, and a `_STATIC_INIT` macro if it can be stack-lived.
  Any caddy fed to `PMIX_THREADSHIFT` needs a `pmix_event_t ev`.
- New wire command → append `PMIX_*_CMD` at the end of the list; never
  renumber; update `pmix_command_string()`.
- Preserve the TMA-aware allocation paths (`pmix_tma_*`) — they let
  objects live in `gds/shmem2` shared memory.
- Treat `pmix_portable_platform_real.h` as read-only vendored code.
- Reach for the `__pmix_attribute_*__` wrappers (defined in
  `pmix_config_bottom.h`) rather than bare `__attribute__`.
