# AGENTS.md: The PINSTALLDIRS `config` Component

`config` is the `pinstalldirs` component that supplies the **build-time
default** installation paths — the `--prefix` and related values chosen
when `configure` ran, compiled straight into the object file. Read the
framework [`AGENTS.md`](../AGENTS.md) first; this file covers only what is
specific to `config`. It is the lowest-priority source (priority **0**),
so it fills in every path field that the higher-priority `env` component
did not override — in a plain installation with no `PMIX_*` environment
variables, that is every field.

## Files

| File | Contents |
|------|----------|
| `pmix_pinstalldirs_config.c` | The component struct; `install_dirs_data` initialized from the `PMIX_*` path macros. |
| `pinstall_dirs.h.in` | Template mapping each `PMIX_*` macro to an Autoconf `@substitution@`. |
| `pinstall_dirs.h` | **Generated** by `configure` from the `.in` template — do not edit or commit. |
| `configure.m4` | Priority (**0**), static compile mode, and `AC_CONFIG_FILES` for the generated header. |
| `Makefile.am` | Builds `libpmix_mca_pinstalldirs_config.la`; lists `pinstall_dirs.h` as a `nodist` generated source. |

## Where the values come from

This is the component the framework doc calls out as reading from a
generated file. The chain is:

1. **`pinstall_dirs.h.in`** defines one macro per path — `PMIX_INSTALL_PREFIX`,
   `PMIX_BINDIR`, `PMIX_LIBDIR`, `PMIX_PKGDATADIR`, and so on — each set
   to an Autoconf substitution token such as `"@prefix@"`, `"@bindir@"`,
   `"@pmixlibdir@"`.
2. **`configure`** expands those tokens (it is listed in this
   component's `configure.m4` via
   `AC_CONFIG_FILES([... src/mca/pinstalldirs/config/pinstall_dirs.h])`)
   and writes the resolved **`pinstall_dirs.h`**. That header is
   generated, `nodist`, and must never be hand-edited — change
   `pinstall_dirs.h.in` instead, then re-run
   `./autogen.pl && ./configure`.
3. **`pmix_pinstalldirs_config.c`** includes the generated header and
   uses the macros as the initializers of its static
   `install_dirs_data`:

   ```c
   .install_dirs_data = {
       .prefix         = PMIX_INSTALL_PREFIX,
       .bindir         = PMIX_BINDIR,
       .libdir         = PMIX_LIBDIR,
       .pmixdatadir    = PMIX_PKGDATADIR,
       ...
   }
   ```

So every field is a compile-time string constant. Note the three
package-scoped fields map through the `PKG` macros:
`pmixdatadir = PMIX_PKGDATADIR`, `pmixlibdir = PMIX_PKGLIBDIR`,
`pmixincludedir = PMIX_PKGINCLUDEDIR`.

Because the substituted values commonly contain unexpanded references
(e.g. `bindir` resolves to `${exec_prefix}/bin`, `exec_prefix` to
`${prefix}`), the raw strings this component provides are **not** final —
the base's `pmix_pinstall_dirs_expand_setup()` pass in `base_init` is
what resolves those `${...}` tokens (and applies `$PMIX_DESTDIR`). See
the framework doc's expansion section.

## The module struct it fills in

`config` provides a `pmix_pinstalldirs_base_component_t` named
`pmix_mca_pinstalldirs_config_component`, opened with
`PMIX_PINSTALLDIRS_BASE_VERSION_1_0_0`. Two things distinguish it from
`env`:

- It is declared **`const`** — its data is fixed at compile time and
  never mutated at runtime.
- Its **`init` hook is absent** (`NULL`): there is nothing to compute at
  runtime, so the base copies `install_dirs_data` directly.

Component open/close functions are `NULL`; the struct closes with
`PMIX_MCA_BASE_COMPONENT_INIT(pmix, pinstalldirs, config)`.

## When it is selected

Always. Like every `pinstalldirs` component it is statically linked
(`NO_DSO`) and always active — there is no selection. Its priority of
**0** makes it the last contributor in the base merge, so it acts as the
backstop that guarantees every path field has a value even when no
`PMIX_*` override is present.

## Gotchas

- **Never edit `pinstall_dirs.h`** — it is regenerated by `configure`
  and your change will be silently discarded. Edit `pinstall_dirs.h.in`,
  and remember that adding or renaming a macro there is a build-system
  change requiring `./autogen.pl && ./configure`.
- The values here are pre-expansion. If a path looks wrong, check
  whether the problem is the `configure`-substituted value in
  `pinstall_dirs.h` or the later expansion/`DESTDIR` step in the base —
  they are different stages.
- The header carries a deliberate `datarootdir=foo` comment line to
  quiet an Autoconf 2.60 warning; leave it in place.
